# PromptWeave 设计说明

## 1. 目标

PromptWeave 是一个面向普通创作者的轻量离线图像提示词工作台。用户输入中文画面描述后，应用会在本地参考库中检索相似模板，输出可直接使用的中文提示词、英文提示词和结构化字段；在配置图像 API 后，还可以直接生成预览图。

首版聚焦三件事：

1. 离线可用，低配置可跑。
2. 支持 GitHub 链接直接导入提示词参考库。
3. 支持提示词优化 API 与图片生成 API 分离配置。

## 2. 产品定位

主定位不是“提示词展示站”，而是“创作工作台”。

默认入口是创作页，左侧保留轻量导航，兼顾日常使用和后续扩展。

## 3. 首版范围

### 3.1 必做

- 本地提示词库管理
- GitHub 链接导入
- Markdown / JSON 解析
- 本地离线检索
- 中文输入 -> 优化提示词
- 中文版 / 英文版 / 双语版输出
- 结构化字段编辑
- GPT Image / Midjourney / Stable Diffusion 三种导出格式
- 提示词优化 API 配置
- 图片生成 API 配置
- 本地历史记录

### 3.2 暂缓

- 账号体系
- 云同步
- 社区分享
- 本地大模型默认集成
- 多工作区协作

## 4. 核心交互

### 4.1 创作流程

1. 用户输入中文自然语言描述。
2. 系统在本地模板库中检索相似案例。
3. 系统抽取结构化字段并生成基础提示词。
4. 用户可在中文提示词、英文提示词、结构化视图之间切换。
5. 如配置提示词优化 API，则进一步润色和本地化。
6. 如配置图片生成 API，则可直接生成预览图并保存本地。
7. 用户可复制、导出、收藏或继续微调。

### 4.2 导入流程

用户粘贴 GitHub URL 即可导入。支持：

- repository URL
- `blob` 文件 URL
- `raw.githubusercontent.com` URL
- README / Markdown / JSON 文件

导入器先预览解析结果，再写入本地库。

## 5. 轻量隔离运行环境

用户提到的“类似 Python 虚拟环境一样”的诉求，首版按三层隔离处理。

### 5.1 应用运行时隔离

- 桌面端使用 Tauri 打包。
- Rust 后端与前端依赖都随应用分发，不依赖全局安装环境。
- 用户无需安装 Python、Node、数据库或额外服务。

### 5.2 工作区隔离

每个 PromptWeave 工作区使用独立目录管理自己的数据，例如：

- 本地提示词库
- 索引文件
- 历史记录
- API 配置
- 缓存
- 导入来源元数据

建议使用类似以下结构：

```text
.promptweave/
  config.json
  cache/
  db.sqlite
  imports/
  exports/
  history/
```

这样每个工作区都像一个独立的“提示词虚拟环境”，可以复制、备份、迁移，不污染系统全局状态。

### 5.3 配置隔离

- API Key 仅保存在本地配置中。
- 默认不启用任何联网能力。
- 优化 API 与图片生成 API 分开配置、分开关闭、分开失效。
- 同步功能默认关闭，只有用户主动开启时才联网。

## 6. 技术方案

### 6.1 推荐栈

- 桌面框架：Tauri
- 前端：React + TypeScript
- UI：Radix UI + Tailwind 或同等轻量组件体系
- 本地存储：SQLite
- 文本检索：SQLite FTS5
- 后端：Rust
- 网络层：可插拔 Provider

### 6.2 为什么不默认用本地大模型

因为首版目标是轻量化和低配置可用。

所以首版采用：

- 本地规则优化
- 词库驱动增强
- 模板结构化填充
- 可选 API 润色

而不是默认内置本地 LLM。

## 7. 模块设计

### 7.1 创作工作台

职责：

- 接收用户中文输入
- 展示优化结果
- 提供复制、导出、收藏、再优化
- 控制图片生成预览

### 7.2 模板库

职责：

- 管理本地提示词合集
- 显示来源、标签、分类、作者、语言
- 支持收藏和编辑
- 支持 FTS 搜索

### 7.3 导入器

职责：

- 识别 GitHub 链接类型
- 拉取 Markdown / JSON
- 自动识别常见 GPT Image 2 prompt 结构
- 导入前预览，导入后去重

### 7.4 优化引擎

职责：

- 抽取中文关键词
- 匹配模板
- 组装结构化字段
- 生成中文完整提示词
- 生成英文版和平台适配版

### 7.5 Provider 层

职责：

- 提示词优化 API
- 图片生成 API
- 自定义兼容接口

Provider 必须是可替换的，不写死某个厂商。

## 8. 数据模型

单条模板建议统一抽象为：

```ts
{
  id: string,
  title: string,
  category: string,
  sourceRepo: string,
  sourceUrl: string,
  sourceLicense?: string,
  author?: string,
  modelHint: "gpt-image-2" | "gpt-image" | "midjourney" | "sd" | "generic",
  language: "en" | "zh" | "mixed",
  promptOriginal: string,
  promptZh?: string,
  promptEn?: string,
  negativePrompt?: string,
  aspectRatio?: string,
  tags: string[],
  previewImageUrls: string[],
  importedAt: string,
  contentHash: string
}
```

## 9. 导出格式

首版内置三种导出格式：

1. GPT Image 风格
2. Midjourney 风格
3. Stable Diffusion / ComfyUI 风格

同一结果支持：

- 中文提示词
- 英文提示词
- 双语版
- 结构化版

## 10. 图像生成

如果用户配置图片生成 API，则创作页右侧显示预览区。

行为要求：

- 未配置时不阻塞提示词生成
- 配置后可直接生成图片
- 支持保存到本地
- 支持记录提示词与图片的对应关系

## 11. 同步策略

首版支持手动导入和手动同步。

后续可扩展：

- 定时同步 GitHub 仓库
- 本地缓存增量更新
- 失败重试

默认不启用自动同步，避免破坏离线和轻量目标。

## 12. 风险与约束

- GitHub 仓库格式不统一，导入器必须容错。
- GPT Image 相关仓库命名不一定统一，模型 ID 不能写死。
- 低配置设备不适合依赖本地大模型。
- 图片生成 API 失败时，提示词输出必须继续可用。

## 13. 迭代顺序

### 第一阶段

- 创作页
- 模板库
- 导入器
- 本地规则优化
- 三种导出格式
- 两个 API 配置

### 第二阶段

- 历史记录增强
- 定时同步
- 更强的翻译策略
- 更细粒度的模板标签体系

## 14. 结论

PromptWeave 的首版原则是：离线优先、轻量优先、创作者优先。

它既要能像“独立工作台”一样打开即用，也要像“虚拟环境”一样把每个工作区、配置和缓存隔离开，避免系统污染和环境依赖问题。
